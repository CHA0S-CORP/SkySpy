## HIGH (3)

### airmet_brief LLM endpoint missing CanUseLLM auth gate and rate throttle
- **File:** `skyspy_django/skyspy/api/aviation.py:446`  ·  **Area:** api_views  ·  **Verifier confidence:** high
- **What's wrong:** The airmet_brief action calls the LLM (aviation_llm.brief_airmet, line 461) but is NOT in _LLM_ACTIONS (only 'explain' is). AviationViewSet defaults to permission_classes=[AllowAny], and get_permissions()/get_throttles() only apply CanUseLLM + ExternalLookupRateThrottle for actions in _LLM_ACTIONS. So airmet_brief runs as AllowAny with no per-action LLM throttle. This is an auth/cost bypass: anonymous visitors on a public deploy can spend LLM tokens via this endpoint, unlike explain/ACARS ai_summary/safety ai_summary which are all gated by CanUseLLM. It is also unthrottled beyond the global anon rate (600/min), so it is an LLM-token abuse vector.
- **Fix:** Proposed fix is correct. Add "airmet_brief" to _LLM_ACTIONS: _LLM_ACTIONS = {"explain", "airmet_brief"}. Verified the DRF @action routing key equals the method name (self.action == "airmet_brief"), matching how "explain" maps to def explain, so both get_permissions() and get_throttles() will then correctly return [CanUseLLM()] and [ExternalLookupRateThrottle()] for this action. No other code change needed. (Mitigations that slightly cap real-world abuse but do not fix the bypass: airmet_brief requires an existing advisory_id — 404 otherwise — and returns {available: False} when the LLM is disabled; but with the LLM enabled on a public deploy, anonymous users can still enumerate advisory IDs and spend tokens, throttled only by the 600/min global anon rate.)

### AlertRuleViewSet over-throttled: entire viewset uses login 'auth' scope (5/min)
- **File:** `skyspy_django/skyspy/api/alerts.py:44`  ·  **Area:** api_views  ·  **Verifier confidence:** high
- **What's wrong:** AlertRuleViewSet sets throttle_classes=[AuthRateThrottle] for the whole viewset. AuthRateThrottle has scope='auth' with rate 5/minute (throttles.py:18,26) — a scope intended for login brute-force protection. This caps ALL alert-rule requests (list, create, toggle, my_rules, bulk_*, export/import) to 5/minute per user, and worse, shares the same 'auth' bucket key with the actual login endpoints, so browsing alert rules consumes the login attempt budget (and vice versa). A dashboard that lists rules plus a couple of toggles will hit HTTP 429 almost immediately.
- **Fix:** Proposed fix is correct in direction. Best fix: follow the existing api/auth.py pattern — replace the class-level `throttle_classes = [AuthRateThrottle]` with a `get_throttles()` that applies AuthRateThrottle only to abuse-prone write/mutating actions (create, update, destroy, bulk_*, import, toggle, test), leaving reads (list, retrieve, my_rules, export) on the global user/anon throttles. Even better, use a dedicated scope (not the login "auth" scope) for alert-rule writes so alert CRUD volume and login brute-force protection don't share a per-user budget. Simply removing throttle_classes entirely (falling back to global 2000/min user throttle) also fixes the functional breakage but drops the write-abuse protection. Note: test settings omit the "auth" scope key so the throttle self-disables in tests (get_rate returns None) — that's why existing tests don't catch this; add a test with the scope configured to prevent regression.

### Scoped API keys are fully privileged over Socket.IO (scope enforcement bypass)
- **File:** `skyspy/socketio/middleware/auth.py:118-158`  ·  **Area:** auth_perms  ·  **Verifier confidence:** high
- **What's wrong:** The Socket.IO API-key validator returns `api_key.user` but never attaches the key's `scopes` anywhere on the connection, and the socket permission layer (`middleware/permissions.py::check_topic_permission` / `check_request_permission` -> `_check_user_permission` -> `profile.has_permission`) checks the OWNING USER's full permission set, not the key's scopes. On the REST side `FeatureBasedPermission`/`HasPermission` read `request.api_key_scopes` to constrain a key to e.g. `aircraft.view` only. Over websockets that constraint is entirely lost: a read-only API key can subscribe to every topic and invoke every mutating request its owner is allowed (create/delete alerts, acknowledge safety, upload audio, etc.). This is a privilege-escalation / scope-bypass for any least-privilege API key used on the realtime channel.
- **Fix:** Proposed fix is correct and matches existing REST semantics. Concretely: (1) change _validate_api_key/authenticate_socket to return the key's scopes alongside the user (e.g. return (user, scopes) or (user, api_key.scopes)); (2) persist scopes in the connect session via sio.save_session in on_connect (add e.g. session['api_key_scopes'] = scopes); JWT/anonymous connects store None (= no scope restriction, full user perms). (3) In BOTH check paths — main.py::_check_request_permission and middleware/permissions.py::_check_user_permission/check_topic_permission — after confirming profile.has_permission(required_perm), additionally require the required permission be covered by the key's scopes when scopes is a non-empty list, reusing auth/permissions.py::_scope_covers_permission (feature.manage_all implies all actions; empty list = all user perms). Note the check must also cover check_topic_permission (topic subscriptions), which currently takes only (user, topic) and has no access to the session — either thread scopes through or move topic/subscription permission checks to a session-aware helper on the namespace like _check_request_permission already is. Also note the bypass is only exploitable when AUTH_MODE is hybrid/private; in AUTH_MODE=public both socket permission checks short-circuit to True so scopes are irrelevant there (and no user-level gating exists either). No change needed for the public-mode path, but the fix should live where hybrid/private is enforced.

## MEDIUM (9)

### AcarsMessage.timestamp uses auto_now_add=True, silently discarding the real message timestamp
- **File:** `skyspy_django/skyspy/models/acars.py:16`  ·  **Area:** acars  ·  **Verifier confidence:** high
- **What's wrong:** The model field is `timestamp = models.DateTimeField(auto_now_add=True, ...)`. With auto_now_add, Django ALWAYS overwrites the value with the row-creation (ingest) time on INSERT and ignores whatever is passed to .create(). But `_store_message` in acars.py carefully parses the upstream message timestamp (epoch/ISO) and passes it as `timestamp=ts`. That value is thrown away. For live UDP this is only slightly wrong, but for the airframes.io firehose poller (messages up to seconds/minutes old, and re-emitted in a rolling window) every stored message is stamped with the poll time, not when the aircraft actually sent it. It also makes the dedupe-by-timestamp and any time-based analytics/ordering reflect ingest cadence, not message time.
- **Fix:** Proposed fix is correct: change models/acars.py:16 to `timestamp = models.DateTimeField(default=django.utils.timezone.now, db_index=True)` (import django.utils.timezone) so the explicitly-passed `timestamp=ts` from _store_message is honored, and generate a migration (this is an editability + default change, not a schema change, but Django still wants a migration for the field definition). Do NOT use `auto_now` (that's for updated-at). If a distinct ingest/received-at time is ever wanted, add a separate `received_at = DateTimeField(auto_now_add=True)` column rather than reusing this one. No caller relies on timestamp==ingest time except the (correct) filtering/ordering in api/acars.py, which actually wants message time — so the fix improves those. Severity lowered to medium: live UDP is only slightly off, impact is concentrated on the airframes.io poller path which is off by default (AIRFRAMES_ACARS_ENABLED=False), and no safety/alert logic depends on this field — it degrades History-tab ordering/analytics accuracy, not correctness of emergency detection.

### Circuit-breaker failure counter TTL is not refreshed on subsequent failures, so sustained-but-slow failures never open the breaker
- **File:** `skyspy_django/skyspy/services/http_client.py:90-105`  ·  **Area:** external_db  ·  **Verifier confidence:** high
- **What's wrong:** `_record_failure` seeds the fail key with `cache.add(key, 1, cooldown)` (TTL=cooldown, default 120s) then uses `cache.incr(key)` on later failures. Django/Redis `incr` does NOT reset the TTL. So the counter expires 120s after the FIRST failure regardless of how many failures follow. A source failing steadily every ~30s (5 failures need ~2min) can perpetually reset before reaching threshold=5, so the breaker never opens and every call keeps paying the full retry budget — defeating the breaker's purpose for exactly the 'hard-down but slow-timeout' upstream it exists to protect against.
- **Fix:** Fix is correct in principle: make the failure window slide by refreshing the TTL on every increment. In the `incr` success path add `cache.touch(key, cooldown)` immediately after `fails = cache.incr(key)` (Django 2.1+ supports touch), or replace the incr with `cache.set(key, fails, cooldown)` after computing `fails = (cache.get(key) or 0) + 1`. Keep the existing `cache.add` first-failure branch (its TTL is already correct) and the ValueError re-seed branch. Apply the touch only in the incr branch — the add branches already set a fresh TTL. Note the re-`set` variant introduces a benign read-modify-write race under concurrent failures (could slightly undercount), so `cache.touch(key, cooldown)` after the atomic `incr` is the cleaner fix. Severity medium is fair: this is a resilience/efficiency degradation (breaker fails to fast-fail, retry budget wasted) rather than incorrect user-facing output or a crash; external calls still ultimately fail-and-raise as before, just without the short-circuit optimization.

### Rich formatter crashes on string numeric fields, aborting the whole dispatch loop
- **File:** `skyspy_django/skyspy/services/rich_formatters.py:168`  ·  **Area:** notifications  ·  **Verifier confidence:** medium
- **What's wrong:** format_alert/format_safety_event format gs, distance_nm and vr with numeric format specs (f"{speed:,.0f} kts", f"{distance:.1f} NM", f"{vr:+,} ft/min") without coercing to a number. ADS-B/cached JSON frequently carries these as strings; f"{'450':,.0f}" raises ValueError. Unlike _format_altitude (which handles strings), these have no guard, and DiscordFormatter/SlackFormatter contain zero except clauses. The dispatcher's _prepare_payload (notification_dispatcher.py:178) calls rich_formatter.format() with no try/except, and dispatch_alert/dispatch_safety_event iterate channels with no per-channel guard (lines 88-96, 133-141) — so one rich (discord/slack) channel with a string speed/vr/distance raises and drops notifications for every remaining channel in that dispatch.
- **Fix:** Fix is valid. Best approach: add small numeric-coercion helpers mirroring _format_altitude and use them for speed/distance/vr in both DiscordFormatter and SlackFormatter (4 call sites total: gs at L168/L403, distance_nm at L179/L412, vr at L277/L564), returning None on non-numeric so the field is simply omitted. e.g. def _fmt_speed(v): try: return f"{float(v):,.0f} kts" except (TypeError, ValueError): return None; guard the append on the result being not None. Also (defense-in-depth, recommended) wrap the rich_formatter.format() call in _prepare_payload (notification_dispatcher.py L178) in try/except logging a warning and leaving rich_payload=None, so any future formatter error can't abort the whole dispatch loop — the plain title/body notification still goes out. The per-formatter coercion is the primary fix; the try/except is the resilience backstop. Note vr uses f"{vr:+,}" which raises a *different* ValueError ("Cannot specify ',' with 's'") on strings but still raises, so it needs coercion too.

### Rich formatting never applied to real safety events (event_type mismatch)
- **File:** `skyspy_django/skyspy/services/rich_formatters.py:622-632`  ·  **Area:** notifications  ·  **Verifier confidence:** high
- **What's wrong:** RichFormatter.format() only recognizes event_type in ('alert') / ('safety','tcas','emergency','proximity'), and EVENT_TYPE_ICONS keys are alert/safety/military/emergency/proximity/tcas. But the actual safety event_type values produced by safety.py are 'squawk_emergency', 'tcas_ra', 'proximity_conflict', 'extreme_vs', 'vs_reversal'. dispatch_safety_event (notification_dispatcher.py:117) passes event_data['event_type'] straight through, so format() returns None for every real safety event and rich Discord/Slack embeds are silently never produced (falls back to plain text); the embed icon is always the default warning glyph.
- **Fix:** Fix is valid. Normalize the raw safety event_type to a rich-format family in BOTH RichFormatter.format() and the EVENT_TYPE_ICONS lookup, or normalize once in dispatch_safety_event before it flows into _prepare_payload. Cleanest: add a helper mapping raw types to buckets — squawk_* -> emergency, tcas_ra -> tcas, proximity_conflict -> proximity, extreme_vs/vs_reversal -> safety — and (a) route to format_safety_event for any of these in format(), and (b) resolve the icon via that normalized family (or just add the raw keys 'squawk_emergency','tcas_ra','proximity_conflict','extreme_vs','vs_reversal' to EVENT_TYPE_ICONS). Note the event-specific 'Vertical Rate' field in format_safety_event checks `event_type == 'tcas'`, so if you normalize tcas_ra->tcas that branch also starts working; if you instead pass the raw type, update that check to include 'tcas_ra'/'vs_reversal'/'extreme_vs'. Do NOT change dispatch_safety_event's event_type used for NotificationTemplate lookup/routing (get_channels_for_priority / get_template_for rely on the specific type) unless you normalize only the value handed to rich_formatter.format — the safest change is confined to rich_formatters.py.

### Airport/navaid/airspace bounding-box omits cos(lat) longitude correction (under-covers at the default feeder latitude)
- **File:** `skyspy/socketio/namespaces/mixins/aviation_data.py:135-143,198-206,240-248`  ·  **Area:** socketio  ·  **Verifier confidence:** high
- **What's wrong:** `_get_airports`, `_get_navaids`, and `_get_airspace_boundaries` compute `lon_delta = radius_nm / 60` — treating a degree of longitude as a degree of latitude. At the shipped default FEEDER_LAT=47.9377 (cos≈0.67) the east/west span of the query box is ~1.5x too narrow, so airports/navaids/airspace east and west of the feeder are silently missing from the map layers. The PIREP query in the same file (`_get_pireps`) correctly divides by `60.0 * max(cos(lat*pi/180), 0.1)`, proving these three are inconsistent and wrong.
- **Fix:** Proposed fix is correct and matches the established _get_pireps idiom. In each of the three handlers replace `lon_delta = radius_nm / 60` with `lon_delta = radius_nm / (60.0 * max(cos(lat * pi / 180), 0.1))` and add `from math import cos, pi` (local import as _get_pireps does). The max(...,0.1) clamp guards against a divide-by-tiny near the poles. Note: _get_metars (L396-397) has the identical missing-correction (bbox string to the weather API) and should be fixed too for consistency, though it wasn't named in the claim. Severity medium is fair: it's silent under-coverage of map reference layers, not a safety/correctness-critical path; higher latitudes make it worse.

### Notification retry loses original title and priority -- critical alerts re-dispatched as generic 'warning'
- **File:** `skyspy_django/skyspy/tasks/notifications.py:202-212`  ·  **Area:** tasks  ·  **Verifier confidence:** high
- **What's wrong:** process_notification_queue re-dispatches failed notifications but hardcodes title='SkysPy Notification' and priority='warning' because the original title/priority were never persisted on NotificationLog. A critical safety alert (e.g. squawk 7700) that fails first delivery and is retried reaches the user as a generic 'warning' titled 'SkysPy Notification' with the real emergency context stripped from the title -- degrading exactly the highest-severity notifications, which are the ones most likely to hit a flaky channel and need retry.
- **Fix:** Fix is valid but can be partially simpler. priority is already recoverable for alert-path notifications: log_entry.details (= alert_data) contains "priority", so the retry could use log_entry.details.get("priority", "info") instead of hardcoding "warning". title, however, is NOT stored anywhere (it's rendered by _render_alert_message and discarded) — so title genuinely needs new persistence. Body/context are already preserved on retry, so the emergency information itself still reaches the user — this is why medium (not critical) is the right severity: the alert is still delivered with correct body, only the title label and severity coloring/urgency are degraded.

### PIREP altitude-band weighting is dead: get_cached_pireps() never returns turbulence_base_ft/turbulence_top_ft keys
- **File:** `skyspy_django/skyspy/services/turbulence.py:160-170`  ·  **Area:** turbulence  ·  **Verifier confidence:** high
